Date: 2026-02-23

- Current technical conclusions from this effort:
  - The original assumption that upstream ring head/tail state handling is fundamentally broken is not supported by current evidence.
  - The ring implementation appears broadly coherent; several bad practices/objective bugs still existed and were fixed.
  - Multiple observed crashes during BCE resume are likely amplified by external async drivers (not VHCI path), especially BCM4377/BRCMFMAC behavior that can destabilize the kernel during the same time window.
  - AAudio also contributed to instability in earlier runs; current safety fixes reduced that failure class, but cannot yet be claimed as a universal root-cause fix.
- Operational interpretation:
  - BCE/VHCI logs in the failing runs are often timing markers of where the system is when failure happens, not definitive proof that BCE logic alone is the primary origin of every reset.
  - Test platform limitation:
  - This MacBook runs a Checkm8-based BridgeOS modification (used to bypass an iCloud lock).
  - The exact BridgeOS changes are unknown and not verifiable with current means.
  - Therefore, suspend/resume findings on this device cannot be treated as fully representative for unmodified T2 MacBook Air/Pro systems.


1) Error #18 from errors.txt
- File: queue_dma.c
- Function: bce_map_segment_list(...)
- Finding:
  The scatter-gather descriptor pointer (`el`) was never advanced inside the
  `for_each_sg(...)` loop. This caused repeated overwrite of the same segment
  entry instead of building a valid descriptor list.
- This is a direct data-path corruption bug in DMA segment list construction.
  Corrupt segment descriptors can break queue/DMA behavior and are consistent
  with severe runtime instability.
- Fix implemented:
  After writing one SG element (`el->addr`, `el->length`), increment `el` so
  each SG entry is stored in the next descriptor slot.

Validation result (after fix #18)
- Evidence source: console photo `documentation/images/51.jpg`.
- Observed behavior:
  - Suspend path completes and reaches deep sleep entry (S3) successfully.
  - Resume progresses through early stages and reaches BCE/VHCI resume.
  - Last visible BCE/VHCI line before failure is:
    `bce_vhci: resume ports`
  - System then hard-resets during/after port resume.
- Current conclusion:
  - Fix #18 appears to resolve the suspend-entry part.
  - Remaining blocker is in the port-resume phase and will be the next target.

Historical hypothesis and update
- Historical hypothesis:

- Implemented:

- Updated conclusion:
  - This removed one clear weakness but did not eliminate the resume reset.

12) Error #25 from errors.txt
- File: `vhci/vhci.c`
- Fix implemented:
  Hardened the resume-endpoint loop in `bce_vhci_bus_resume()`:
  - iterate only active ports (`1..port_count`);
  - read mapped `devid` once and validate bounds before dereference;
  - validate `vhci->devices[devid]` is non-NULL before using `tq_mask`/queues.
- Why this is an objective fix:
  Removes unchecked pointer/index dereference in a critical PM path and turns
  stale mapping cases into explicit log+skip behavior instead of hard crash.
- Scope note:
  This is a correctness/safety hardening fix; it may reduce crash vectors but is
  not yet proof of the primary root cause on its own.
